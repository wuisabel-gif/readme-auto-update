# README writing policy

## Project README

Prefer this order when supported by evidence:

1. One-sentence purpose
2. Who or what the project is for
3. Key capabilities
4. Architecture or implementation shape
5. Installation and minimal usage
6. Current status and limitations
7. Contributing, security, and license links

Do not turn a project README into a commit log. Recent work should update the description of the
current system, not dominate it.

## Profile README

Prefer a compact, problem-led introduction followed by only the categories that exist:

1. A human explanation of what connects the work, when evidence supports one
2. A small highlights section describing why each project is useful
3. Organization or collaboration work with concrete contribution evidence
4. Open-source contributions
5. A themed catalog only when enough projects exist
6. Anonymous private-work activity when enabled

Rank repositories by meaningful recent contribution evidence, then repository signals such as
stars and recency. Treat counts as selection evidence, not as the main story. Avoid exhaustive
catalogs unless requested.

## Evidence language

- State repository descriptions and observed architecture as facts.
- Explain the problem a project solves when its README, description, or documentation supports it.
- Never invent a first-person motivation. If motivation is not documented, describe behavior and
  usefulness directly.
- Label a next step as an inference unless a roadmap, issue, or explicit user instruction supports
  it.
- Use “contributed to” when scope is unclear; do not imply sole authorship.
- Do not equate commit count with impact.
- Do not expose private identity through contextual clues after names have been redacted.

## Style

Use direct prose, short sections, concrete nouns, varied sentence rhythm, and descriptive links.
Let project descriptions sound like explanations from a builder: what was frustrating, what the
tool changes, and where it fits. Avoid inflated language, generic enthusiasm, fake quotes,
unsupported metrics, repetitive sentence templates, and long technology badge walls. Match the
existing voice when manual README content establishes one.
